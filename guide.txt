  └─ Fields: maintenance_approval_threshold, monthly_dues_amount, late_fee_percentage

TIER 2: USERS & ROLES
──────────────────────

✓ users
  └─ 4 roles: admin, chairman, assistant, member
  └─ Fields: id, email, password_hash, full_name, phone_number, role, plaza_id, unit_id
  └─ unit_id: Only populated for members (residents). Which apartment?
  └─ Relationships: belongs to plaza, belongs to unit (if member)

──────────────────────────

✓ units

TIER 4: MAINTENANCE & FINANCIAL OPERATIONS
────────────────────────────────────────────

✓ maintenance_posts

  └─ Relationships: belongs to plaza, created by user, approved by user

  └─ Key: linked_assessment_id (if funded by special assessment)

✓ maintenance_attachments ⭐ NEW TABLE

  └─ Fields: id, maintenance_post_id, file_url, file_type, file_name, file_size_bytes

  └─ NEW: file_type (image/jpeg, video/mp4, application/pdf, etc.)
  └─ NEW: file_size_bytes (track storage usage)
  └─ Benefits:
     ├─ Clear separation of concerns
     ├─ Can query by file type
     ├─ Audit trail (who uploaded what)
     ├─ Storage tracking per user
     ├─ Easy to add file expiry policies
     └─ Follows relational database best practices

✓ special_assessments

  └─ Fields: id, title, reason, required_amount, shortfall, occupied_units, per_unit_amount
  └─ Status: PENDING, APPROVED, REJECTED, APPROVED_AND_IMPLEMENTED

✓ payments

  └─ Fields: id, user_id, action, resource_type, resource_id, before_state, after_state
  └─ Key: before_state, after_state (JSON snapshots - what changed?)

  └─ Relationships: belongs to user who performed action

✓ transaction_logs ⭐ NEW TABLE
  └─ Financial transaction log (DEDICATED FINANCIAL AUDIT TRAIL)
  └─ Fields: id, transaction_type, amount, description, balance_before, balance_after
  └─ NEW: Separate from audit_logs for financial integrity
  └─ transaction_type: 'credit' (money in) or 'debit' (money out)
  └─ NEW: balance_before & balance_after (proves financial consistency)
  └─ NEW: related_resource_id & related_resource_type (links to maintenance, payment, etc.)
  └─ Benefits:
     ├─ Financial audits separate from action audits
     ├─ Easy to verify: "How did balance go from 245K to 232.5K?"
     ├─ Can reconcile: sum(credits) - sum(debits) = current_balance
     ├─ Compliance-ready for financial reporting
     ├─ Prevents accidental balance inconsistencies
     └─ Full trail of every rupee in/out

================================================================================
                          ATOMIC TRANSACTIONS (Critical)
================================================================================

When these operations happen, EVERYTHING must succeed or EVERYTHING rolls back:

SPECIAL ASSESSMENT APPROVAL (Most Complex):
  When Chairman approves special assessment for Rs. 17,500 (Rs. 973 per unit):

  The system ATOMICALLY:
  1. Updates ALL 18 occupied units' "due" field by +973
  2. Creates notification record for each resident
  3. Queues high-priority FCM pushes for all 18
  4. Updates special_assessment.status → APPROVED_AND_IMPLEMENTED

  If step 1 updates 17 units but step 2 fails on unit 18:
  → ENTIRE operation rolls back. No units are updated. No notifications created.

  This ensures consistency: either all residents are assessed OR none are.

MAINTENANCE POST CREATION (with cost > threshold):
  When Admin logs maintenance for Rs. 180,000:

  The system ATOMICALLY:
  1. Validates cost > 0 and cost <= master_pool_balance
  2. Deducts Rs. 180,000 from master_pool_balance
  3. Creates maintenance_posts record
  4. Creates transaction_log entry (debit: 180,000)
  5. Creates audit_log entry (action: create_maintenance)

  If step 2 fails (balance check):
  → ENTIRE operation rolls back. Balance unchanged. Post not created.

  This ensures: deduction, post creation, and logs happen together or not at all.

PAYMENT PROCESSING:
  When resident pays Rs. 5,000:

  The system ATOMICALLY:
  1. Updates units.due from 5,000 → 0
  2. Updates units.status from "Pending" → "Paid"
  3. Sets units.last_payment_date timestamp
  4. Creates payment record
  5. Creates transaction_log entry (credit: 5,000)
  6. Creates audit_log entry (action: log_payment)
  7. Queues FCM notification

  If any step fails:
  → ENTIRE operation rolls back. Unit state unchanged.

  This ensures: payment recorded completely or not at all.

FILE ATTACHMENT CREATION:
  When Admin uploads photo for maintenance:

  The system:
  1. Uploads file to CDN (S3/Firebase) → gets URL back
  2. Creates maintenance_attachments record with file_url
  3. Creates audit_log entry

  If CDN upload fails: don't create database record
  If database creation fails: notify admin, file stays on CDN (cleanup job removes it)

================================================================================
                        SEPARATE TABLES FOR ATTACHMENTS
================================================================================

WHY SEPARATE TABLES FOR FILES?

✗ ANTI-PATTERN (Don't do this):
  maintenance_posts {
    id,
    title,
    cost,
    image_urls: ['url1', 'url2', 'url3']  ← JSON array (bad)
    photo_count: 3                         ← Denormalized count (bad)
  }

✓ BEST PRACTICE (Do this):
  maintenance_posts {
    id,
    title,
    cost,
    attachments: has_many maintenance_attachments
  }

  maintenance_attachments {
    id,
    maintenance_post_id (foreign key),
    file_url,
    file_type (mime type: image/jpeg, video/mp4, etc.)
    file_name,
    file_size_bytes (track storage),
    uploaded_by (who uploaded - audit trail),
    uploaded_at
  }

BENEFITS OF SEPARATE TABLES:
  ├─ Clear separation of concerns
  ├─ Audit trail (who uploaded what file?)
  ├─ Query flexibility:
  │  ├─ "Find all PDFs uploaded by this user"
  │  ├─ "Show me total storage used per plaza"
  │  ├─ "How many video attachments in tickets?"
  │  └─ "List all files uploaded by User X on Date Y"
  ├─ Storage tracking (which files use most space?)
  ├─ Relational integrity (proper foreign keys)
  ├─ Follows database normalization rules
  ├─ Scales better with thousands of attachments
  ├─ Easy to add file expiry policies
  ├─ Easy to delete files and clean up metadata
  └─ One attachment per row (simpler logic)

WHERE APPLIED:
  ├─ maintenance_posts ← maintenance_attachments
  └─ tickets ← ticket_attachments

================================================================================
                        FINANCIAL AUDIT PATTERN
================================================================================

TWO SEPARATE AUDIT TABLES:

1. AUDIT_LOGS (What action happened?)
   └─ Tracks: who did what, when, with before/after snapshots
   └─ Example: "Admin created maintenance post #123"

2. TRANSACTION_LOGS (How did money move?)
   └─ Tracks: every credit and debit with balance snapshots
   └─ Example: "Debit Rs. 12,500, balance 245,000 → 232,500"

WHY TWO TABLES?

✗ OLD WAY (Problematic):
  All financial changes mixed into audit_logs:
  └─ Hard to query: "What financial transactions happened?"
  └─ Hard to reconcile: "Prove balance changed correctly"
  └─ Compliance issues: "Show financial audit trail"

✓ NEW WAY (Best Practice):
  Separate transaction_logs for finances:
  └─ Easy to query: SELECT * FROM transaction_logs WHERE plaza_id = X
  └─ Easy to reconcile: sum(credits) - sum(debits) = balance
  └─ Compliance: "Here's every rupee in/out"

EXAMPLE FLOW:

Admin logs maintenance for Rs. 12,500:

  AUDIT_LOG created:
    ├─ action: 'create_maintenance'
    ├─ resource_type: 'maintenance'
    ├─ resource_id: 'maint-123'
    ├─ user_id: admin-xyz
    ├─ before_state: {cost: null, status: null}
    ├─ after_state: {cost: 12500, status: 'IMPLEMENTED'}
    └─ created_at: 2024-03-18T10:00:00Z

  TRANSACTION_LOG created:
    ├─ transaction_type: 'debit'
    ├─ amount: 12500
    ├─ description: 'Maintenance: Fixed hallway pipe'
    ├─ related_resource_type: 'maintenance_post'
    ├─ related_resource_id: 'maint-123'
    ├─ balance_before: 245000
    ├─ balance_after: 232500 ✓ (proof)
    ├─ recorded_by: admin-xyz
    └─ created_at: 2024-03-18T10:00:00Z

AUDIT QUESTIONS ANSWERED:
  ├─ Q: "What actions happened in plaza on March 18?"
  │  A: SELECT * FROM audit_logs WHERE plaza_id = X AND created_at >= '2024-03-18'

  ├─ Q: "Prove balance went from 245K to 232.5K"
  │  A: SELECT * FROM transaction_logs WHERE balance_before = 245000 AND balance_after = 232500

  ├─ Q: "What was the balance 30 days ago?"
  │  A: SELECT balance_before FROM transaction_logs ORDER BY created_at DESC LIMIT 30

  ├─ Q: "How much money came in/out last month?"
  │  A: SELECT SUM(IF(transaction_type='credit',amount,0)) as income,
           SUM(IF(transaction_type='debit',amount,0)) as expenses
       FROM transaction_logs WHERE plaza_id = X AND created_at >= '2024-02-18'

  └─ Q: "Who approved this maintenance expense?"
     A: SELECT * FROM audit_logs WHERE resource_id = 'maint-123' AND action = 'approve_maintenance'

================================================================================
                          FILE STORAGE ARCHITECTURE
================================================================================

Database stores: URLs only (strings)
  └─ file_url: "https://cdn.s3.amazonaws.com/plaza-123/maint-456/image1.jpg"

Actual files stored: On CDN (S3, Firebase, Cloudinary, etc.)
  ├─ Never store binary files in database
  ├─ Database tracks metadata only:
  │  ├─ file_url (CDN URL)
  │  ├─ file_type (mime type for queries)
  │  ├─ file_size_bytes (track storage)
  │  ├─ file_name (original name)
  │  └─ uploaded_by (audit trail)
  └─ When you need to delete: query attachments, delete from CDN, delete record

WHY THIS ARCHITECTURE?
  ├─ Database bloat avoided (no binary data)
  ├─ File access faster from CDN (geo-distributed)
  ├─ Can serve images from multiple regions
  ├─ Can version, archive, or delete files independently
  ├─ Metadata stored in DB for queries
  ├─ Actual files on CDN (infinitely scalable)
  ├─ Cheaper storage (S3 is cheaper than database)
  └─ Easy to implement cache invalidation

WORKFLOW:
  1. Admin uploads photo for maintenance
  2. File goes to CDN → gets URL back
  3. maintenance_attachments.file_url = that URL
  4. Database stores URL + metadata (type, size, uploader)
  5. When resident views maintenance: load image from CDN URL
  6. When admin deletes maintenance: delete from CDN, delete record from DB

================================================================================
                          RELATIONSHIP EXAMPLES

One Maintenance Post → Many Attachments:
  maintenance_posts: 1 ──→ ∞ maintenance_attachments
  Example: "Fixed hallway pipe" has 3 photos (before, during, after)

One Ticket → Many Attachments:
  tickets: 1 ──→ ∞ ticket_attachments
  Example: Complaint about water damage has 2 photos

One Unit → Many Payments (over months/years):
  units: 1 ──→ ∞ payments
  Example: Apt 301 pays Rs. 5,000 every month = 12 payments/year

One User → Many Device Tokens (phone, tablet, web):
  users: 1 ──→ ∞ device_tokens
  Example: Hammad logged in on iPhone + iPad = 2 tokens

One Plaza → Many Transaction Logs (financial movements):
  plazas: 1 ──→ ∞ transaction_logs
  Example: Every maintenance/payment/assessment creates a transaction log

One Plaza → Many Audit Logs (all actions):
  plazas: 1 ──→ ∞ audit_logs
  Example: Every login/create/approve action logged

One Special Assessment → Many Notifications (one per resident):
  special_assessments: 1 ──→ ∞ notifications
  Example: Generator assessment triggers 18 notifications (one per occupied unit)

One Ticket → One Maintenance Post (optional):
  tickets: 1 → 0 or 1 → maintenance_posts
  Example: Complaint about leaking pipe may be converted to maintenance work
